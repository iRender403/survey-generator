### 进行数据埋点

| 数据埋点   | 埋点方式 | 埋点原因 |
| :----- | :--- | :--- |
| Layout | 如下   | 如下   |

```tsx
  const comRef = useRef(null);
  const renderCount = useRef(0);

  // 浅比较
  let isSame = comRef.current === singleSelectStatues;
  /**
   * 数据埋点
   */
  console.log({
    渲染次数: ++renderCount.current,
    上次数据: comRef.current,
    这次数据: singleSelectStatues,
    引用相同: isSame,
    渲染原因: isSame ? "父组件导致的" : "Redux数据变化导致的",
  });

  useEffect(() => {
    comRef.current = singleSelectStatues;
  });

```

为什么要数据埋点：数据验证直觉

为什么要这么设计：引发Layout组件重新渲染的原因有两个，一个是父组件渲染带动子组件渲染，另外一个是useSelector依赖变化导致的重新渲染

这里仅仅追踪useSelector的依赖是否发生了变化，从而实现对于渲染原因的跟踪

<br />

为什么使用ref : ref可以将数据跨渲染存储,从而实现先前数据和现在的数据的比较

为什么要进行比较：这涉及组件重新渲染的逻辑：

什么时候会重新渲染：

| 导致页面重新渲染的情况        | 重新渲染的原理                                                                                                  |
| :----------------- | :------------------------------------------------------------------------------------------------------- |
| 父组件渲染导致的子组件渲染      | React 的渲染是递归的：当父组件重新渲染时，默认会触发所有子组件的重新渲染（除非子组件通过 React.memo 、 PureComponent 或 shouldComponentUpdate 进行优化） |
| Hooks中的依赖发生变化产生的渲染 | useState 、 useReducer 的 dispatch、 useSelector 、 useContext 等 Hook 的依赖值发生变化时，会触发该组件的重新渲染                  |
| 组件自身状态变化           | 调用 setState 或 dispatch 后，React 会安排该组件重新渲染                                                                |
| Context 值变化        | 当 Context.Provider 的 value 变化时，所有消费该 Context 的组件都会重新渲染                                                   |
| 强制更新               | 调用 forceUpdate() （类组件）或使用 useReducer 的 hack 强制刷新                                                         |
