> 对于组件市场实现的反思

## 两种实现方案对比

### 1. 组件市场(`materialView)`：路由驱动方案

**实现思路**：通过嵌套路由 `/material/select/single-select` 来区分不同组件，使用 `loader` 将 URL 参数同步到 Redux。

```typescript
// router/index.tsx
{
  path: 'select',
  children: [
    { path: 'single-select', element: <SingleSelect />, loader: capturePathLoader },
    { path: 'multi-select', element: <MultiSelect />, loader: capturePathLoader },
  ]
}
```

**问题**：

- 路由仅作为"参数传递工具"，实际状态仍存储在 Redux
- 形成"路由 → Redux → 组件"的绕路
- 需要 `useOutletContext` 进行父子传参
- 嵌套层级深，配置复杂

### 2. Redux + Map 映射方案

**实现思路**：将 `currentSelectStatus` 作为 Redux 状态，通过 Map 映射动态渲染对应组件。

```typescript
// schemaSlice.ts
const initialState = {
  currentSelectStatus: 'single-select',  // 当前选中组件类型
  com: { /* 各组件状态 */ }
};

// 组件渲染
const Component = componentMap[currentSelectStatus];
```

**优势**：

- 单一状态源头，无需 URL 解析
- 任何组件可直接通过 `useSelector` 访问状态
- 无需 `useContext` 或 `useOutletContext` 传参
- 路由配置大幅简化

## 反思：为什么可以去掉路由？

**核心判断标准**：状态是否需要被 URL 承载？

| 场景                       | 是否需要 URL            | 结论      |
| ------------------------ | ------------------- | ------- |
| MaterialView 的"当前选中组件"   | ❌ 无需分享/刷新恢复,并且页面嵌套深 | 用 Redux |
| 组件市场的分类导航 (select/input) | ✅ 需要分享链接            | 保留路由    |

**结论**：MaterialView 的组件切换是纯粹的界面级临时状态，没有分享价值，使用 Redux 更干净。

## 模式抽象

> Redux + Map = 动态组件渲染系统

在一定程度上可以简化路由的嵌套层级，减少参数传递的复杂度，但是对于简单的路由来说，这样配置其实是更加复杂一些的，所以简单的路由使用，不涉及复杂的路由嵌套的时候，可以使用这个方案

<br />

1. 简单页面切换&#x20;
   1. 使用方向 : 路由
   2. 原因 : 配置简单React Router 成熟&#x20;
2. deeply nested 嵌套（3层+）&#x20;
   1. Redux + Map&#x20;
   2. 减少嵌套地狱 同一组件需要不同传参方式 Redux + Map 统一从 Redux 取，避免 props drilling 需要频繁切换、
3. 状态复杂&#x20;
   1. Redux + Map&#x20;
   2. 避免 URL 同步开销
4. 普通 CRUD 后台系统&#x20;
   1. 路由&#x20;
   2. 足够简单，维护成本低

<br />

## 对模式抽象的验证：需要改进和进一步权衡

| **场景特点**                    | <br />            | <br />                    | <br />                    |
| :-------------------------- | :---------------- | :------------------------ | :------------------------ |
| <br />                      | **核心痛点**          | **推荐方案**                  | **注意事项**                  |
| **重展示、需分享/书签** (如商品详情、大类目)  | 状态需持久化、SEO/分析追踪   | **扁平路由 (Path)**           | 结合按需加载优化首屏                |
| **同层级轻量级切换** (如 Tab 页、组件预览) | 避免路由嵌套地狱，需支持浏览器后退 | **Query Params + Map**    | 避免深层嵌套，保持 URL 的轻量         |
| **高度复杂的内部交互** (如多步表单、画布工具)  | 状态复杂、无需分享、不需要后退   | **Redux / Context + Map** | 做好页面销毁时的状态清理 (Unmount 卸载) |
| **庞大的组件集映射** (如组件市场)        | 首屏包体积爆炸           | **状态管理 + 懒加载 Map**        | 必须配合 `React.lazy` 实现按需加载  |
